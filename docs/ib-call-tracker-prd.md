# 大投行电话会追踪与观点提炼产品方案

## 1. 项目定位

目标不是做一个“研报内容搬运器”，而是做一个面向交易者的 **投行观点 intelligence layer**：

- 持续追踪 `金十海外研报` 这类会汇总高盛、JPM、PB 周报、内部电话会、Desk color 的研究源
- 自动识别其中哪些内容属于“投行观点更新”
- 把长摘要拆成可交易的结构化信息
- 让用户一眼看到：
  - 哪家投行在说什么
  - 看多/看空什么
  - 背后的逻辑链条是什么
  - 观点相较过去有什么变化
  - 哪些资产和板块最值得跟踪

一句话定义：

> 把零散、非结构化、带平台编辑提炼痕迹的投行电话会摘要，转成“可追踪、可比较、可预警、可交易”的观点数据库。

## 2. 为什么这个方向成立

以 `金十海外研报` 当前聚合的海外研究内容看，源里已经有比较稳定的模式：

- 高盛内部电话会摘要
- JPM 金属与矿业电话会摘要
- PB 周报、客户 flow、仓位变化整理
- 大宗、利率、股票、AI/软件、地缘政治等跨资产讨论

这类内容对交易者有价值，但现在存在 4 个问题：

1. 信息是长摘要，阅读成本高
2. 不同条目之间不可比较，难看出“观点变化”
3. 条目混有编辑层解释，原始观点与二次解读容易混在一起
4. 很多内容是“可交易”的，但没有被抽成结构化卡片

所以真正的产品机会，不在“摘要一次”，而在：

- 持续跟踪
- 结构化归档
- 变化检测
- 信号分发

## 3. 最值得做的产品形态

我建议不要一上来做很重的终端，而是按三层做：

### 方案 A：追踪简报工具

形态：

- 每天/每周抓取金十海外研报新增内容
- 自动判断是否属于投行 call / PB note / desk color
- 输出中文重点摘要

优点：

- 最快上线
- 验证用户有没有持续阅读需求

缺点：

- 护城河弱
- 很容易沦为内容聚合

### 方案 B：投行观点雷达

形态：

- 不只给摘要，还给结构化观点卡
- 支持“按银行 / 资产 / 主题 / 地区 / 时间”筛选
- 支持同一资产的多家机构观点对比
- 支持观点变化提醒

优点：

- 产品感最强
- 交易者最容易形成使用习惯
- 后续最容易做 alert、回测和订阅

缺点：

- 需要更强的数据建模和提取质量

### 方案 C：交易 Copilot

形态：

- 用户直接问：“最近大行怎么看铜、油、英债、软件股？”
- 系统自动汇总近 7/30 天所有相关电话会观点

优点：

- 交互自然
- 非常适合深度用户

缺点：

- 如果底层没有结构化数据库，回答会很虚

## 4. 推荐路线

推荐顺序是：

1. 先做 `方案 B`
2. 用 `方案 A` 作为内容分发层
3. 最后再叠加 `方案 C`

原因很简单：

- 纯摘要没有长期壁垒
- 纯聊天没有结构化底座会不稳定
- “观点雷达”是最能沉淀资产的中间层

## 5. MVP 应该长什么样

### 5.1 核心页面

MVP 做 4 个页面就够了：

#### 1. Feed

类似研究流，但每条不是普通内容，而是“结构化观点卡”。

每张卡至少包含：

- 来源渠道
- 原始发布时间
- 银行 / desk / 类型
- 3 行摘要
- 核心观点方向
- 涉及资产
- 时间维度
- 置信度
- 原文链接 / 研报页链接

#### 2. Bank View Board

把各家投行近期观点做成看板。

示例维度：

- Goldman Sachs
- JPMorgan
- Morgan Stanley
- BofA
- GS PB / Prime / Flow desk

每家下方展示：

- 最近 7 天新增观点
- 主要看多资产
- 主要看空资产
- 本周观点变化

#### 3. Asset Lens

按资产聚合，比如：

- 铜
- 原油
- 英债前端
- 黄金
- 软件股
- 医疗、公用事业

每个资产页显示：

- 近 30 天谁在提
- 看多/看空分布
- 支撑逻辑
- 风险点
- 最近变化

#### 4. Alerts

让用户订阅：

- 某家银行
- 某个资产
- 某类主题

例如：

- “高盛提到英国前端利率时提醒我”
- “出现铜观点下调时提醒我”
- “PB flow 里如果出现金融股大规模净卖出，推送给我”

## 6. 真正的产品壁垒：不是摘要，而是“观点图谱”

要把系统分成三层：

### 第一层：原始内容层

- 金十海外研报条目
- 原始标题与摘要
- 机构名 / 栏目名 / 发布时间
- 必要时补充外链原始研究或新闻稿

### 第二层：结构化观点层

从内容中抽出：

- 哪家机构
- 哪个 desk
- 哪个资产
- 方向是什么
- 时间维度是什么
- 催化剂是什么
- 风险/反例是什么

### 第三层：变化与关系层

这是最有价值的部分：

- 同一家银行对同一资产是否发生转向
- 不同银行在同一主题上是否分歧扩大
- 某个主题是否从“宏观逻辑”传导到“行业/个股逻辑”
- 某个观点是否在后续多次被强化

也就是说，用户想看的不是“这条帖讲了什么”，而是：

> 过去两周，关于铜的 sell-side 观点，是不是在系统性转弱？

## 7. 关键功能设计

### 7.1 智能分类

先判断一条帖到底是什么：

- 投行电话会摘要
- PB / flow / positioning note
- 采访/播客摘要
- 作者个人观点
- 快讯型转发

如果分类错了，后面全会错。

### 7.2 观点抽取

把长文拆成标准字段：

- `institution`
- `asset`
- `direction`
- `time_horizon`
- `driver`
- `risk`
- `trade_expression`
- `confidence`

### 7.3 变化检测

例如系统自动发现：

- 一周前 `JPM: neutral copper`
- 本周 `JPM: underweight copper`

则生成：

- `view_change = downgrade`

### 7.4 观点分歧检测

例如：

- GS 看多英国前端
- MS 中性
- JPM 偏谨慎

则在资产页直接显示：

- `consensus = mixed`

### 7.5 可交易表达

这是最重要的功能之一。

原文里往往是叙事，你要把它转成交易者能直接用的表达：

- 方向：看多 / 看空 / 中性
- 标的：资产 / 板块 / 风格 / 地区
- 逻辑：为什么
- 催化剂：什么条件会兑现
- 失效条件：什么情况下观点失效

## 8. 数据流程设计

推荐的数据流水线：

### Step 1. 数据采集

优先级：

1. 拉取金十海外研报指定页的海外研究条目
2. 保存标题、摘要、发布时间、机构、链接、主题标签
3. 必要时补抓原始研报或外链长文

### Step 2. 候选内容识别

通过规则 + LLM 分类识别：

- 是否提到机构名
- 是否提到电话会/周报/PB/note/desk 等词
- 是否存在编号式长摘要

### Step 3. 结构化抽取

把候选内容转成统一 schema。

### Step 4. 去重与归并

有些内容是同一主题的续发或连续更新，要归并成一个 event。

### Step 5. 变化检测

同银行 + 同资产 + 同 desk 历史对比，识别升级/降级/转向。

### Step 6. 生成用户层产物

- feed 卡片
- alert
- 周报
- 资产页
- 问答接口

## 9. 用户真正会买单的展示方式

### 9.1 卡片化展示

建议每条观点卡只展示 6 个东西：

- 谁说的
- 说了什么
- 看多/看空什么
- 为什么
- 对应交易表达
- 和上次相比变了没有

### 9.2 热力图 / 矩阵

很适合交易者：

横轴：

- 资产

纵轴：

- 机构

颜色：

- 看多 / 中性 / 看空

这样用户能一眼看到分歧和共识。

### 9.3 时间线

把同一资产上各家观点按时间串起来。

这比单独看一条摘要有用得多。

### 9.4 “一句话结论 + 可展开原文”

交易者通常先看结论，再决定要不要读细节。

所以默认展示：

- `TL;DR`
- `Top 3 Takeaways`
- `Trade Expression`

展开后再看完整摘要。

## 10. 最关键的产品原则

### 原则 1：严格区分“原始机构观点”和“平台二次提炼”

这个非常重要。

像 `金十海外研报` 这类聚合源的内容很有价值，但很多时候是：

- 机构信息
- 平台编辑提炼
- 系统归纳判断

三者混在一起。

所以系统里必须有字段：

- `source_layer = direct / summarized / interpreted`

否则用户会把二手加工内容误认为是一手 sell-side 原话。

### 原则 2：必须保留可追溯性

每张卡都要能回到：

- 金十海外研报原始条目
- 对应摘要页或后台记录
- 外链原始研究

否则专业用户不会信。

### 原则 3：先做人机协同，不要一开始就全自动

MVP 阶段建议：

- LLM 先抽取
- 人工只 review 高价值卡片

因为金融语义里：

- “看多 long-end vol”
- “逢低做多前端利率”
- “股票防御配 + 能源对冲”

这些表达很容易被抽错。

## 11. 技术实现建议

如果你做 web 产品，我建议用下面这套：

- 前端：Next.js
- 后端：Node.js / Python 任一
- 数据库：Postgres
- 搜索：Postgres FTS + 向量检索
- 队列：Redis / BullMQ
- 定时抓取：Cron / worker
- 模型层：
  - 分类模型：识别是否为投行观点摘要
  - 抽取模型：抽结构化字段
  - 对比模型：检测观点变化

### 最小表结构

建议至少有这些表：

- `sources`
- `posts`
- `events`
- `institutions`
- `assets`
- `view_items`
- `alerts`
- `daily_digests`

其中最核心的是 `events` 和 `view_items`。

## 12. 你最容易踩的坑

### 坑 1：只盯一个平台

如果产品完全依赖一个平台，会有很大上游依赖风险。

更稳的做法：

- 第一阶段以 `金十海外研报` 为主源
- 第二阶段扩到 5-20 个同类高质量研究源
- 第三阶段再扩到 newsletter / Substack / podcast summary

### 坑 2：摘要做得很好，但没有“变化”

用户第一周会觉得很新鲜，第二周就疲劳。

真正让人持续打开的是：

- 今天哪些观点变了
- 哪些资产开始出现一致看法
- 哪些地方分歧突然变大

### 坑 3：全部自动化，精度不够

金融内容一旦错抽方向，产品可信度会很伤。

建议高价值内容先做：

- 自动抽取
- 人工确认
- 再沉淀成训练样本

### 坑 4：只做内容，不做交易映射

交易者不想只看到“叙事”，他想知道：

- 该盯什么
- 用什么表达交易
- 风险在哪

## 13. 商业化方向

最自然的收费点有三个：

### 1. 高级 Alert

- 资产级提醒
- 银行级提醒
- 观点变化提醒
- 大宗 / 外汇 / 利率 / 美股板块提醒

### 2. 深度数据库

- 历史观点查询
- 观点转向记录
- 多机构比较
- 导出和 API

### 3. 研究工作台

- 自建 watchlist
- 自定义日报
- 团队协作标注

## 15. 一个很实用的首页原型

首页可以直接做成四块：

### Top strip

- 今日新增观点数
- 今日涉及机构数
- 今日高频资产
- 今日观点变化数

### Main feed

- 最新观点卡

### Right panel

- 今日最大变化
- 今日最一致看多
- 今日最一致看空

### Bottom heatmap

- 机构 x 资产 的观点矩阵

这样打开首页，交易者 30 秒内就知道今天值不值得看。

## 17. 我对这个项目的判断

我认为这个方向能做，但前提是你要把产品定义清楚：

- 不是“研报内容摘要工具”
- 不是“宏观新闻聚合器”
- 而是“卖方观点跟踪与交易映射系统”

如果定义对了，这个产品最有价值的不是内容本身，而是三件事：

1. 观点结构化
2. 观点变化检测
3. 观点到交易表达的映射
